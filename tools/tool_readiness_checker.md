# Tool Readiness Checker Mission\n\nPurpose: Given a proposed tool idea, check whether the Tool Foundry already has the needed capabilities, whether a new tool is needed, what risk level it has, whether it requires approval, and what the next action should be.\n\nInputs:\n- raw_idea: required string containing the proposed tool idea.\n- mode: optional quick, standard, or deep; default standard.\n- registry_snapshot: optional current registry list.\n- user_constraints: optional privacy, budget, approval, or deployment constraints.\n\nOutputs:\n- decision: use_existing_tool, create_new_tool, revise_existing_tool, needs_owner_approval, or do_not_build.\n- existing_capabilities: matching or partial tools with fit_level and reason.\n- new_tool_needed: boolean.\n- recommended_tool_name: string or null.\n- risk_level: low, medium, or high.\n- approval_required: boolean.\n- approval_reasons: list.\n- next_action: plain-English next step.\n- mission_outline: complete mission outline when a new tool is recommended.\n- safety_notes: list.\n- confidence: low, medium, or high.\n\nBehavior:\n1. Compare the proposed idea against the provided registry snapshot.\n2. Treat approved exact or strong capability matches as usable existing tools.\n3. Recommend a new tool only when no approved tool covers the needed capability.\n4. Flag owner approval for public deployment, paid API use, storing personal/sensitive data, sending emails/messages, publishing, external accounts, real-world actions, permission increases, or autonomous scheduled actions.\n5. Use conservative risk scoring when intent is unclear.\n6. Never ask the user to code, debug, upload files, choose frameworks, or redeploy manually.\n\nSuccess criteria:\n- Finds approved existing capabilities when present.\n- Avoids duplicate tool builds.\n- Produces clear risk and approval decisions.\n- Produces a complete mission outline for new tools.\n- Gives plain-English next action.\n\nTest cases:\n- Startup idea analysis with approved idea_analyzer present returns use_existing_tool.\n- Automatic customer email sender returns approval_required true.\n- PDF invoice extractor without exact approved extractor returns create_new_tool.\n- Paid public scraper returns approval_required true.\n- Vague input returns low confidence with cautious next action.\n
